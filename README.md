# まじかる単位認定（旧題: ルール推理計算ゲーム）

仕様書v2（`ルール推理ゲーム_仕様書_v2.md`。v1からの変更点は文書冒頭にまとめてあります）に基づく実装です。`server`（Node.js + Express + Socket.io）と `client`（React + Vite）の2つのプロジェクトで構成されています。

## v2での主な変更点

- 予測フェイズの計算式入力が、テキスト入力からソフトウェアキーパッド（電卓風）に変更されました。
- ルールプールが15個に全面改訂され、段階A（数値の前処理）/B（演算子の意味変更）/C（結果の後処理）に加えて、新しく段階D（見た目の演出のみで数値には影響しない）が追加されました。段階Dには「ナベアツ風の読み上げ演出」と「猫（結果がすべて2の並びのとき、数字の代わりに猫の画像を表示）」の2つがあります。
- 回答フェイズの判定が4段階（CLEAR/B/C/不正解）から、学校の成績風の5段階（卒業／優／良／可／不可）に変更されました。
- 回答フェイズの見た目が「解答用紙」風（マークシート風の丸チェックボックス、判定結果は赤い判子のスタンプ演出＋効果音）に変更されました。
- 全体の見た目をパステル×モダン、ゴシック系フォントのテイストに一新しました。

## その後の追加機能

- **あきらめるボタン**: 卒業判定の回答画面に「あきらめる」ボタンを追加しました。押すと確認ダイアログの後、判定「中退」として結果画面に遷移し、「君には失望した。」というメッセージとともに正解ルールが公開されます（他の不正解と違い、このラウンドはそこで終了します）。
- **CPU（一人プレイ）**: ロビー画面でホストがCPUを親役として追加できるようになりました。CPUは自分では監視画面などを使わず、親に指名された瞬間にランダムに2〜3個のルールを選んで即座に試験問題フェイズを開始します。一人で部屋を作ってCPUを追加すれば、そのまま一人プレイが成立します。
- **タイトルのグラフィック化・UI全体の装飾強化**: トップ画面のタイトルをSVGのグラフィック（グラデーション文字＋キラキラ装飾）に変更し、見出し用に装飾フォント「Yusei Magic」を追加しました。Tabler Iconsを各画面の見出しやボタンに幅広く追加し、背景にも控えめな星模様のテクスチャを敷いています。

## セットアップ・起動方法

### 1. サーバー

```
cd server
npm install
npm start
```

デフォルトで `http://localhost:3001` で起動します。

### 2. クライアント

```
cd client
npm install
npm run dev
```

デフォルトで `http://localhost:5173` で起動します（Viteの開発サーバー）。

同じPC以外の端末から参加させる場合は、クライアントの環境変数 `VITE_SERVER_URL` にサーバーのアドレス（例: `http://192.168.x.x:3001`）を設定してください。

```
VITE_SERVER_URL=http://192.168.x.x:3001 npm run dev -- --host
```

## 実装時に補った設計上の前提（仕様書に明記がなかった細部）

以下は仕様書のレビューの中では詰め切れなかった細部について、実装のためにこちらで決めた挙動です。プレイしてみて違和感があれば調整してください。

- **演算子の優先順位**: 括弧なし・四則演算のみという仕様のもと、常に式を左から右へ順番に計算します（× ÷ を先に計算する、という一般的な優先順位も、右から左に計算するルールも採用していません。後者はv2で仕様として明示的に廃止されています）。
- **段階Bの演算子の意味変更**: 演算子の意味は、式に元々書かれている記号（+ − × ÷）だけを見て一度だけ解決します。ルール適用によって記号の意味が変わっても、その変わった意味に対してさらに別のルールが連鎖して適用されることはありません。
- **段階Aの最小/最大ルールのタイ（同値）処理**: 式の中に同じ値の項が複数ある場合（例: `3+3+5` の最小値3が2つ）、対象のルールはタイになっている項すべてに適用されます。
- **ルール適用によって割る数が0になった場合**: 入力時点では0除算でなくても、ルール適用の結果として割る数が0になることがあります。この場合は「計算不能（÷0）」という結果として全員に表示し、手番は通常通り消費します。
- **割り切れない場合の表示**: 小数第2位までを四捨五入した小数で表示します（循環小数を正確に表示する機能は実装していません）。
- **段階Cの桁操作ルール（鏡反転・絶対値・2倍）**: 結果が負の数や小数になった場合、整数部分の絶対値に対して桁操作を適用し、符号・小数部分は別途保持しています。
- **段階D（ナベアツ・猫）の優先順位**: 計算結果が「3の倍数または3を含む」と「すべて2の並び（22, 222, 2222…）」の両方の条件を同時に満たす場合（例: 222）は、猫の演出が優先されます。
- **回答フェイズの5段階判定**: 数・内容とも完全一致→卒業、数のみ一致し正解ルールを含まない→可、数は不一致だが正解ルールを部分的に含む→良、数は一致し正解ルールを部分的に含む→優、それ以外→不可、としています。「正解ルールを全部含みつつ余分にも選んでいる」場合（例: 正解が[1,2]で子が[1,2,3,4]を選択）も「数は不一致・正解ルールを含む」＝良に含めています。
- **回答フェイズで卒業に至らなかった場合の他の子への公開範囲**: 選んだルールの内容や判定結果は本人にのみ表示し、他の参加者には公開していません（何を推理して外れたかが他の子のヒントにならないようにするためです）。
- **「ホスト」と「親」の分離**: 部屋を作成した人（ホスト）は部屋の管理権限（解散・親の指名）を持つ固定の役割、「親」はホストがラウンドごとに指名する役割として分離しています。ホスト自身が親を兼任することも可能です。
- **部屋の定員**: ホストを含めて合計6人（最大）としています。
- **ホスト・親のいずれかが切断した場合**: ホストが切断した場合は部屋を解散します。進行中の親（ホスト以外）が切断した場合や、予測フェイズのターン順に含まれるメンバーが切断した場合は、ロビー画面に戻して親の再指名からやり直せるようにしています。
- **親専用の計算内訳（trace）表示**: 監視画面（ParentMonitor）の履歴テーブルに「内訳（あなただけに表示）」列を追加し、実際にどのルールがどの数値・演算子をどう変えたかを日本語の短文で表示しています。子には送信されません（サーバー側で親以外の履歴からtraceフィールドを取り除いています）。

## 猫の画像・スタンプ効果音について

- `client/public/game-assets/cat.jpg` … 「猫」ルール成立時に表示される画像。ご提供いただいた写真を、猫全体が収まるように正方形にトリミング済みです（表示時はCSSで自動的に丸くなります）。
- `client/public/game-assets/stamp.mp3` … 判定発表時のスタンプ音。ご用意いただいた和太鼓の音源に差し替え済みです。

（`public`フォルダに置いているのは、Viteのビルドでこのフォルダの中身がそのままサイトのルート直下にコピーされ、`src`配下からの相対パス参照より確実に読み込めるためです。）

どちらもこのzipに同梱済みのため、追加の作業は不要です。なお、これらのファイルが万が一見つからない場合でもアプリは問題なく動作します（猫は絵文字🐱、効果音は無音で自動的に代替されます）。

## 動作確認について

このプロジェクトを作成した環境ではネットワークの制限により `npm install`（npmレジストリへのアクセス）が実行できず、実際にサーバー・クライアントを起動しての動作確認は行えていません。代わりに以下を確認済みです。

- ルールエンジン（`ruleEngine.js`）は外部依存なしの純粋なロジックのため、`node --test src/ruleEngine.test.js` で単体テストを実行し、15ルールそれぞれの動作例と段階Dの優先順位、および万・億の位までの読み上げを含む24パターンすべてが期待通りの結果になることを確認しました。
- 5段階判定ロジック（`gameLogic.js`）は直接呼び出すシミュレーションで、卒業/優/良/可/不可のすべてのパターンが期待通りになることを確認しました。
- 全ファイルについて、UTF-8としての整合性・括弧やブレースの対応・Node.jsの構文チェック（`node --check`、JSXファイルは対応チェック）を行い、問題がないことを確認しました。

お手元の環境で `npm install` を実行した後、実際にサーバー・クライアントを起動しての通し確認をお願いします。手順は以下の「お手元での動作確認手順」を参照してください。

## お手元での動作確認手順

### 1. Node.jsをインストールする（まだの場合）

1. https://nodejs.org/ja にアクセスし、「LTS」と書かれている方のインストーラーをダウンロードして実行する（Windows/Mac共通）。
2. インストール後、ターミナル（Windowsは「コマンドプロンプト」または「PowerShell」、Macは「ターミナル」アプリ）を開き、次のコマンドを実行してバージョンが表示されればインストール成功。
   ```
   node -v
   npm -v
   ```

### 2. プロジェクトを展開する

1. お渡しした `rule-guess-game.zip` をダブルクリックして展開する（Windowsは右クリック→「すべて展開」でも可）。
2. 展開してできた `rule-guess-game` フォルダの場所を確認しておく（例: デスクトップに置いた場合は `Desktop/rule-guess-game`）。

### 3. サーバーを起動する

1. ターミナルを開き、`cd` コマンドでフォルダの中の `server` に移動する。展開先がデスクトップなら例えば以下のようになる。
   ```
   cd Desktop/rule-guess-game/server
   ```
   （フォルダをターミナルにドラッグ＆ドロップすると自動でパスが入力される場合もあります）
2. 必要なパッケージをインストールする（初回のみ、少し時間がかかります）。
   ```
   npm install
   ```
3. サーバーを起動する。
   ```
   npm start
   ```
   「rule-guess-game server listening on port 3001」のような表示が出れば起動成功。このターミナルはゲーム中ずっと開いたままにしておく。

### 4. クライアント（画面）を起動する

1. 別のターミナルウィンドウを新しく開く（3.のターミナルはそのまま残しておく）。
2. 今度は `client` フォルダに移動する。
   ```
   cd Desktop/rule-guess-game/client
   ```
3. パッケージをインストールする（初回のみ）。
   ```
   npm install
   ```
4. 開発サーバーを起動する。
   ```
   npm run dev
   ```
5. 表示された `http://localhost:5173` のようなURLをブラウザで開く。

### 5. 実際に遊んでみる

1. ブラウザのタブをもう1つ（もしくは別のブラウザ）開き、同じ `http://localhost:5173` にアクセスする。1つ目のタブで「部屋を作る（ホスト）」、2つ目のタブで「部屋に入る（参加者）」を選び、発行された部屋コードを使って参加する。
2. ロビー画面でホストが「〇〇さんを親にする」ボタンを押してこのラウンドの親を指名すると、親側の画面がルール選択画面に切り替わる。ホスト自身を親に指名することも可能。
3. 同じPC内であれば、この方法（複数タブ）で1人でも動作確認ができる。
4. 実際にスマホなど別の端末から参加させたい場合は、README上部の「同じPC以外の端末から参加させる場合」の手順（`VITE_SERVER_URL` の設定）を参照してください。

### うまく動かないときは

- `npm install` でエラーが出る場合は、Node.jsのバージョンが古い可能性があります。`node -v` で v18以上であることを確認してください。
- ブラウザで画面が表示されない場合は、サーバー（3.）とクライアント（4.）の両方のターミナルが起動したままになっているか確認してください。

## Renderへの無料デプロイ（同じネットワーク外の人も参加できるようにする）

このプロジェクトはRenderの無料Webサービスにそのままデプロイできるように、以下の対応済みです。

- サーバー（`server/src/index.js`）が、クライアントのビルド成果物（`client/dist`）を自分で配信するようになっている（ローカル開発時はこの機能は働かず、今まで通り `run-server.bat`/`run-client.bat` が使えます）。
- クライアント（`client/src/socket.js`）は、本番ビルド時は自動的に「自分自身と同じアドレス」のサーバーに接続するようになっている（開発時の挙動は変更なし）。
- ルートの `package.json` と `render.yaml` により、Render側の「ビルドコマンド」「起動コマンド」の設定を自動化している。

**注意**: 無料枠は15分操作がないとスリープし、次のアクセス時に起動まで30〜60秒かかります。スリープ中は誰も接続できず、進行中の部屋のデータ（メモリ保持のみ）は消えます。身内で遊ぶ分にはこれで十分ですが、この制約は把握しておいてください。

### 1. GitHubにコードを置く

1. GitHubで新しいリポジトリを作成する（例: `rule-guess-game`、Public/Privateどちらでも可）。README等の初期ファイルは追加しない（空のリポジトリを作る）。
2. お手元の `rule-guess-game` フォルダで、ターミナルを開いて以下を実行する。
   ```
   cd rule-guess-game
   git init
   git add .
   git commit -m "initial commit"
   git branch -M main
   git remote add origin https://github.com/【あなたのユーザー名】/rule-guess-game.git
   git push -u origin main
   ```
   （既にGitHub連携済みの場合は、`git add .` → `git commit -m "v2"` → `git push` だけで反映されます。）

### 2. Renderでデプロイする

1. https://render.com にアクセスし、GitHubアカウントでサインアップ/ログインする。
2. ダッシュボードで「New +」→「Blueprint」を選び、先ほどpushしたリポジトリを連携する。リポジトリ内の `render.yaml` が自動的に読み込まれ、ビルド/起動コマンドが設定された状態でサービスが作成される。
   - Blueprintが使えない/表示されない場合は、「New +」→「Web Service」から手動で連携し、以下を設定する。
     - Build Command: `npm install && npm run build`
     - Start Command: `npm start`
     - Instance Type: Free
3. 「Deploy」を実行する。初回ビルドは数分かかる。
4. デプロイが完了すると `https://rule-guess-game-xxxx.onrender.com` のようなURLが発行される。このURLを家族・参加者に共有すれば、ネットワークの外からでも参加できる。

### 3. コードを更新したとき

`rule-guess-game` フォルダ内でファイルを変更した後、以下を実行すればRender側に自動反映される（Renderがリポジトリの更新を検知して自動的に再デプロイする設定になっている）。

```
git add .
git commit -m "update"
git push
```
